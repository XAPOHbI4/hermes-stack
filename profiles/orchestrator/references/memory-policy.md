# Memory policy — что хранить, что никогда, и где чему место

Грузится по триггеру «сохранить факт / что класть в память / куда положить знание». Дополняет trust-pyramid из `source-map.md`.

## Сохранять в profile memory (MEMORY.md)
- стабильные предпочтения владельца (стиль, формат ответа, таймзона);
- устойчивые рабочие правила/решения, переживающие сессии;
- указатели «где лежит правда» (а не саму правду, если она меняется).

## НИКОГДА не сохранять в память
- секреты/токены/ключи/credentials — `.env`/секрет-хранилище, не в md/память/chat;
- прогресс задач / временное состояние → Kanban/reports;
- сырые логи → reports;
- клиентские/личные данные сверх стабильного owner-context;
- факты о ТЕКУЩЕМ состоянии системы (что запущено/сколько/жив ли) → это live-check, память устаревает.

## Что куда класть (what-goes-where)
| Тип знания | Куда | Пример |
|---|---|---|
| Стабильные правила/предпочтения | profile memory | стиль, таймзона, «не делать X» |
| Текущие решения/архитектура | wiki/ (если есть) или contract | «сейчас роутим так» |
| Статичные контракты/правила | `references/` | proof policy, source-map |
| Повторяемая процедура | `skills/` | как делать задачу-N |
| Прогресс/состояние задач | Kanban | статус, blocked, deps |
| Доказательства работы | reports/closures/reviews | proof, логи проверок |
| Секреты | `.env`/секрет-store | токены |
| Старые диалоги | session search | только recall, не истина |

## Гигиена
- **Сжимать, не дописывать:** держать MEMORY.md компактным (крон-консолидатор, лимит ~2200–3500 симв).
- Перед сохранением — проверить, нет ли дубля; править существующий файл, а не плодить новый.
- Удалять то, что оказалось неверным.
- Не сохранять то, что и так в коде/гите/контракте.

## Приоритет (из trust-pyramid)
Память **проигрывает** live state и source of truth. Перед утверждением о «сейчас» — live-check, а не recall из памяти. Полная иерархия — `references/source-map.md`.
