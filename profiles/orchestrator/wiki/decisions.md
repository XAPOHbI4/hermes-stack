# Ключевые решения (и почему)

Лог решений по поведению системы. Новое решение → дописать сюда (что + почему).

- **Engine-first decompose** (Kanban `--triage`), а не «агент сам зовёт architect-тул». Почему: soft-промпт-подход провалился 3/3 (модель забывала дёрнуть тул). Дробление гарантирует платформа; модель решает роутинг/сложность. *Принцип: думает модель — исполняет движок.*
- **Тонкое ядро AGENTS.md (≤12КБ) + references по требованию.** Почему: большие контракты съедают окно и размывают приоритеты.
- **Адверсариальный proof-gate** (отдельный Claude-ревьюер). Почему: не закрывать на доверии; self-report → REWORK по умолчанию.
- **Proof policy: тип доказательства = тип задачи.** behavior-changing → ≥smoke с логом запуска, не document. Почему: «отчёт» ≠ подтверждение, что правило работает. Enforce ревьюером, не хардкод-гейт (остаёмся model-driven).
- **`pid not alive` ≠ провал.** Воркер = дочерний процесс шлюза; рестарт шлюза его SIGKILL-ит, ретраи восстанавливают. Метрики отделяют process-death от task-failure. Рестарт — graceful (`gw-restart`, SIGUSR1 drain), не `systemctl restart`.
- **Лимиты = окна, не деньги.** Claude — oauth usage (5h/7d); Codex — rate_limits в session-rollout. Мониторятся, алерт при ≥80%.
- **claude_run без plan-mode.** Сырой `--permission-mode plan` уходит в ExitPlanMode-петлю → виснет.
- **Trust pyramid.** Память и старые чаты не бьют live state / source of truth. Перед утверждением о «сейчас» — live-check.
- **Мониторинг — внешние systemd-таймеры, не cron в шлюзе.** Сторож не должен умирать вместе со шлюзом.
- **Обучение — человеко-читаемый слой** (eval-дайджест + уроки Claude) поверх стокового GEPA; применение выводов — осознанно, не авто-мутация контрактов.
- **Воспроизводимость.** Слой версионируется в git (`hermes-stack`), накатывается `install.sh`; секреты/state — вне репо.
