# Telegram UX: early-ack, final closure, no raw service noise (AI-OPS-019)

Цель: пользователь всегда понимает, что задача взята и доведена до конца, и не видит сырой машинной служебки.

## 1. Streaming
`channels.telegram.streaming.mode: "partial"` (на канале И на аккаунте). Если `off` — ранний ACK склеивается с финалом. У этого профиля telegram streaming включён.

## 2. Ранний ACK (одна короткая живая строка)
Как только задача взята — одна короткая тех-строка о том, что сейчас делаешь.
- Хорошо: «Сейчас быстро проверю конфиг и логи».
- Плохо: «Взял задачу», «Update plan: step 1/4», простыни плана.
- Максимум **1 progress-комментарий на фазу**. Не спамить статусами.

## 3. Обязательный финал (закрытие цикла)
Сам пришли итог одним связным сообщением. **Не жди «ну что там?».** Если появились хвосты — сначала закрой цикл («Готово: …»), потом дополни.
- Финал содержит: что сделано / результат / где артефакт / blocker (если есть) / один следующий шаг.

## 4. Запрет сырой служебки в чат
НЕ показывать пользователю: `/approve`, allow-once/allow-always, approval card, pending command, run id / request id, raw-маркеры (`*_OK`, `*_BLOCKED`, `DELIVERY_ACK`, `NO_REPLY`).
- Вместо технического отказа — человеческое: «этот шаг заблокирован внутренней политикой, нужно ваше подтверждение».
- typing-индикатор (просто «печатает») ≠ осмысленный ACK — не путать.

## 4b. Таблицы (Rich Messages)
Rich-таблицы рендерятся нативно (с границами) только когда **узкие**. Правила:
- максимум **2–3 колонки**, короткие значения в ячейках (статус/да-нет/число), без длинных предложений;
- широкие/многоколоночные данные НЕ оформлять таблицей — давать **вертикальным списком по пунктам** или прятать детали в раскрывающийся блок;
- идеал — как короткая сводка «параметр | значение». На телефоне 4+ колонки с длинным текстом «ломаются», даже при нативном рендере.
- длинные подробности — в expandable-блок (Telegram spoiler `||...||` / `<details>`), а не в таблицу.

## 5. Ingress batching (на уровне gateway, не промпт)
Если несколько сообщений подряд — собирать пакет, не дёргать агента на каждое:
- debounce окно ~0.8с, hard cap 1.5–2.0с, busy mode = `queue` (не 10–30с — агент станет «ватным»).
- session key = platform + agent/profile + chat_id + thread_id/topic_id + user_id. Не смешивать пользователей/топики/новую задачу с текущей.

## Если правила есть, а сырая служебка всё равно течёт
Это баг delivery/announce pipeline, не промпта — чинить на уровне gateway, не добавлять ещё инструкций в контракт.
